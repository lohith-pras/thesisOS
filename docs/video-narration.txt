Supervisor feedback is usually written like this: strengthen the literature review, compare an approach, or revise a thesis section. It rarely repeats the exact title of a paper. ThesisOS turns that ordinary feedback into a reviewable, evidence-backed research trail.

I built ThesisOS with Codex. Codex helped shape the local-first architecture, implement and debug the Zotero connector, design the semantic retrieval evaluation, close the approval-gated evidence loop, and generate the automated judge-mode capture you are seeing. The main Codex build session is included in the submission.

The workflow starts by preserving the supervisor's original wording. Codex CLI can decompose messy feedback into a validated task graph using the user's local Codex login, so basic use does not require API credits. Every proposed task begins pending. A rejected literature task cannot search the library. Approval authorizes only the read-only search; it does not authorize a note or filesystem write.

For retrieval, ThesisOS connects to personal or group Zotero libraries through the local read-only API. It indexes titles, authors, venues, abstracts, Zotero tags, and DOI. Local Ollama embeddings provide semantic similarity, while metadata matching supplies an explainable second signal. The interface reports its retrieval mode, relevance threshold, abstract coverage, and why each result matched. Unchanged paper embeddings are cached locally. On five supervisor-style evaluation queries against a real forty-paper group library, the baseline reaches point eight three recall at five and a mean reciprocal rank of one.

The researcher then selects the papers that truly count as evidence. Only those stable Zotero source IDs move forward to the dedicated Evidence notes step. Codex CLI drafts the grounded literature note through the authenticated local session. Before processing, the user sees and approves the boundary. Codex returns strict structured output, and every citation must match a selected source ID. Unknown citations are rejected. If Codex is unavailable, ThesisOS visibly falls back to a deterministic grounded template; GPT-5.6 remains an optional API adapter.

The final note is always previewed. In judge mode, ThesisOS stops here and cannot write to the filesystem. In the full local workflow, writing to an Obsidian vault requires a separate approval and existing files are never overwritten.

ThesisOS is designed for students who need help moving research forward without giving up judgment, provenance, or control. The agent does the organizing and retrieval work. The researcher decides what counts as evidence and what is allowed to change.
